기존 초성퀴즈·훈민정음 봇에 “유물 시스템”을 추가해 주세요.

현재 유저 포인트와 게임 데이터는 SQL DB로 관리됩니다.
기존 초성퀴즈와 훈민정음의 기본 게임 로직은 이미 구현되어 있다고 가정합니다.
이번 작업에서는 기존 게임 로직을 크게 바꾸지 말고, **보상 정산 단계에 유물 보너스·콤보 보너스·랜덤 콤보를 연결**하고, 별도로 **유물 강화·성급강화·배틀·랭킹 기능**을 추가해 주세요.

---

# 전체 개발 목표

추가할 기능은 다음입니다.

1. 유저별 유물 보유
2. 유물 정보 조회
3. 일반 강화
4. 성급강화
5. 유물 보너스 점수 적용
6. 일반 콤보 적용
7. 랜덤 콤보 적용
8. 유물 배틀
9. 유물 랭킹
10. 강화·성급강화·배틀 로그 저장
11. SQL DB 트랜잭션 처리
12. 포인트 음수 방지 및 동시성 문제 방지

# 작업 단위 1. SQL DB 테이블 추가/수정

## 1-1. 유저 기본 데이터 확인

기존 유저 테이블에 다음 컬럼이 있는지 확인하고, 없으면 추가해 주세요.

필요 컬럼:

* user_id
* point
* total_earned_point
* total_spent_point
* created_at
* updated_at

주의:

* point는 절대 음수가 되면 안 됩니다.
* 포인트 차감·증가가 발생하는 작업은 SQL transaction으로 처리해 주세요.

---

## 1-2. 유물 보유 테이블 생성

유저당 대표 유물 1개를 보유하는 구조로 구현해 주세요.

필요 컬럼:

* user_id
* artifact_id
* artifact_name
* star
* level
* created_at
* updated_at

기본값:

* 신규 유저 또는 유물이 없는 유저는 1성 1강 유물을 지급합니다.
* star 기본값: 1
* level 기본값: 1

## 1-4. 보너스 소수점 누적 컬럼 또는 테이블 생성

유물 보너스는 퍼센트 방식이므로 소수점 보너스를 누적해야 합니다.

필요 컬럼:

* user_id
* bonus_remainder

예시:

기본 보상 50P, 보너스율 1.1%일 경우 보너스는 0.55P입니다.
이 값을 버리지 말고 bonus_remainder에 누적합니다.
bonus_remainder가 1 이상이 되면 정수 부분만 실제 포인트로 지급하고, 나머지 소수점은 다시 저장합니다.

# 작업 단위 2. 유물 기본 생성 로직

기본 유물:

* level: 1
* artifact_name: 임의의 1성 유물 이름

유물 이름은 일단 임시값으로 처리해도 됩니다.
예: “신비한 나침반”

완료 기준:

* 유물이 없는 유저가 `/유물정보`를 입력하면 1성 1강 유물이 생성되고 정보가 표시되어야 합니다.

# 작업 단위 3. 일반 강화 기능 구현

명령어 예시:

* `/유물강화`

일반 강화는 현재 성급 안에서 강화 수치를 올리는 기능입니다.

## 일반 강화 기본 규칙

* 성공: level + 1
* 유지: level 변화 없음
* 실패: level - 1
* 단, 최소 level은 1입니다.
* 1강에서 실패해도 1강을 유지합니다.
* 일반 강화 실패로 유물은 파괴되지 않습니다.
* 포인트는 성공/유지/실패와 관계없이 강화 시도 시 차감됩니다.
* 현재 성급의 최대 강화 수치에 도달하면 일반 강화는 불가능합니다.

성급별 최대 강화:

* 1성: 50강
* 2성: 80강
* 3성: 100강

---

## 일반 강화 비용 파라미터

비용은 “현재 강화 수치 n에서 다음 강화로 갈 때” 적용합니다.

예:

* 1성 1강 → 2강 비용은 n = 1 기준
* 1성 49강 → 50강 비용은 n = 49 기준

### 1성 강화 비용

* 1강~10강: `5 × n`
* 11강~20강: `50 + 5 × (n - 10)`
* 21강~30강: `100 + 8 × (n - 20)`
* 31강~40강: `180 + 12 × (n - 30)`
* 41강~49강: `300 + 20 × (n - 40)`

예시:

* 1강 → 2강: 5P
* 10강 → 11강: 50P
* 20강 → 21강: 100P
* 30강 → 31강: 180P
* 40강 → 41강: 300P
* 49강 → 50강: 480P

### 2성 강화 비용

* 1강~20강: `100 + 10 × n`
* 21강~40강: `300 + 35 × (n - 20)`
* 41강~60강: `1000 + 70 × (n - 40)`
* 61강~70강: `2400 + 120 × (n - 60)`
* 71강~79강: `3600 + 200 × (n - 70)`

예시:

* 1강 → 2강: 110P
* 20강 → 21강: 300P
* 40강 → 41강: 1000P
* 60강 → 61강: 2400P
* 79강 → 80강: 5400P

### 3성 강화 비용

* 1강~20강: `700 + 35 × n`
* 21강~40강: `1400 + 80 × (n - 20)`
* 41강~60강: `3000 + 160 × (n - 40)`
* 61강~80강: `6200 + 300 × (n - 60)`
* 81강~90강: `12200 + 500 × (n - 80)`
* 91강~99강: `17200 + 800 × (n - 90)`

예시:

* 1강 → 2강: 735P
* 20강 → 21강: 1400P
* 40강 → 41강: 3000P
* 60강 → 61강: 6200P
* 80강 → 81강: 12200P
* 99강 → 100강: 24400P

---

## 일반 강화 확률 파라미터

확률은 성공, 유지, 실패의 합이 100%가 되도록 처리합니다.

### 1성 강화 확률

* 1강~5강: 성공 75%, 유지 23%, 실패 2%
* 6강~10강: 성공 70%, 유지 25%, 실패 5%
* 11강~20강: 성공 64%, 유지 27%, 실패 9%
* 21강~30강: 성공 58%, 유지 29%, 실패 13%
* 31강~40강: 성공 52%, 유지 30%, 실패 18%
* 41강~45강: 성공 47%, 유지 29%, 실패 24%
* 46강~49강: 성공 42%, 유지 20%, 실패 38%

### 2성 강화 확률

* 1강~20강: 성공 76%, 유지 20%, 실패 4%
* 21강~40강: 성공 68%, 유지 24%, 실패 8%
* 41강~60강: 성공 60%, 유지 27%, 실패 13%
* 61강~70강: 성공 53%, 유지 27%, 실패 20%
* 71강~79강: 성공 47%, 유지 25%, 실패 28%

### 3성 강화 확률

* 1강~20강: 성공 70%, 유지 24%, 실패 6%
* 21강~40강: 성공 62%, 유지 27%, 실패 11%
* 41강~60강: 성공 54%, 유지 30%, 실패 16%
* 61강~80강: 성공 47%, 유지 31%, 실패 22%
* 81강~90강: 성공 41%, 유지 30%, 실패 29%
* 91강~99강: 성공 36%, 유지 29%, 실패 35%

---

## 일반 강화 처리 순서

1. 유저 조회
2. 유물 조회
3. 유물이 없으면 1성 1강 기본 유물 생성
4. 현재 star, level 확인
5. 현재 성급의 최대 강화 상태인지 확인
6. 최대 강화라면 일반 강화 중단 후 성급강화 안내
7. 현재 level 기준 강화 비용 계산
8. 유저 point가 비용보다 적으면 중단
9. SQL transaction 시작
10. 유저 포인트 차감
11. 강화 결과 랜덤 판정
12. 성공이면 level + 1
13. 유지면 level 유지
14. 실패면 level - 1, 단 최소 1강
15. 유물 정보 업데이트
16. 유물 통계 업데이트
17. 포인트 로그 저장
18. 강화 로그 저장
19. transaction commit
20. 결과 메시지 출력

완료 기준:

* 포인트 부족 시 강화가 진행되지 않아야 합니다.
* 최대 강화 상태에서는 일반 강화가 진행되지 않아야 합니다.
* 실패 시 level이 0 이하로 내려가면 안 됩니다.
* 포인트와 유물 상태 변경은 하나의 transaction으로 처리되어야 합니다.

# 작업 단위 4. 성급강화 기능 구현

보상 유물 강화 수치 확률:

* 30강: 18%
* 31강: 16%
* 32강: 14%
* 33강: 12%

* 35강: 9%
* 36강: 7%
* 37강: 6%

* 54강: 10%
* 55강: 9%

* 58강: 5%

## 3성 성급강화

* 3성은 최종 성급입니다.
* 3성은 성급강화가 불가능합니다.

---

## 성급강화 처리 순서

1. 유저 조회
2. 유물 조회
3. 유물이 없으면 1성 1강 기본 유물 생성
4. 현재 star, level 확인
5. star = 3이면 성급강화 불가
6. 1성 50강 또는 2성 80강 조건 확인
7. 조건 미달이면 중단
8. 성급강화 비용 확인
9. 포인트 부족 여부 확인
10. SQL transaction 시작
11. 포인트 차감
12. 성공/실패 랜덤 판정
13. 성공 시 다음 성급 1강으로 변경
14. 실패 시 기존 유물 파괴 처리
15. 실패 보상 유물 생성
16. 성급강화 통계 업데이트
17. 포인트 로그 저장
18. 성급강화 로그 저장
19. transaction commit
20. 결과 메시지 출력

완료 기준:

* 1성 50강이 아니면 2성 도전 불가
* 2성 80강이 아니면 3성 도전 불가
* 3성은 성급강화 불가
* 실패 시 반드시 보상 유물이 지급되어야 함
* 포인트 차감과 유물 변경은 하나의 transaction으로 처리되어야 함

# 작업 단위 6. 유물 보너스 점수 적용

기존 초성퀴즈·훈민정음의 보상 정산 단계에 유물 보너스를 추가해 주세요.

## 유물 보너스 파라미터

1성:

* 기본 보너스: 1%
* 강화 1강당 추가 보너스: 0.1%
* 최대 보너스: 6%

2성:

* 기본 보너스: 8%
* 강화 1강당 추가 보너스: 0.1%
* 최대 보너스: 16%

3성:

* 기본 보너스: 18%
* 강화 1강당 추가 보너스: 0.17%
* 최대 보너스: 35%

예시:

* 1성 1강: 1.1%
* 1성 50강: 6%
* 2성 1강: 8.1%
* 2성 80강: 16%
* 3성 1강: 약 18.17%
* 3성 100강: 35%

---

## 보너스 소수점 누적 방식

보상 계산 시 소수점 보너스는 버리지 말고 누적합니다.

처리 예시:

1. baseReward = 50P
2. artifactBonusRate = 1.1%
3. bonus = 0.55P
4. 정수 보너스 0P는 즉시 지급
5. 소수점 0.55P는 bonus_remainder에 누적
6. bonus_remainder가 1 이상이면 정수 부분만 포인트로 지급
7. 남은 소수점은 계속 저장

완료 기준:

* bonus_remainder는 유저별로 저장되어야 합니다.

# 작업 단위 8. 훈민정음 보상 정산 연결

훈민정음 보상 계산:

* 우승 보너스 = 정답 수 × 30P, 최대 250P
* baseReward = 일반 정답 보상 + 우승 보너스

우승자가 아닌 유저:

* 일반 정답 보상만 받음
* 우승 보너스 없음

추가 처리:

1. 기존 로직으로 baseReward 계산
2. 유저 유물 조회
3. 유물 보너스율 계산
4. 일반 콤보 판정
5. 랜덤 콤보 판정
6. 적용할 콤보 보너스율 계산
7. 최종 지급 포인트 계산
8. bonus_remainder 반영
9. 포인트 지급
10. total_earned_point 갱신
11. 포인트 로그 저장

완료 기준:

* 훈민정음 기존 보상 구조는 유지
* 단어 정답 보상은 한 판 최대 150P 제한
* 우승 보너스는 최대 250P

# 작업 단위 9. 일반 콤보 기능 구현

일반 콤보는 유저의 승리 횟수에 따라 발동합니다.

## 일반 콤보 파라미터

* 콤보 목표값: 3~6회 사이 랜덤
* 초성퀴즈 정답 시 승리 1회로 계산
* 훈민정음 최종 우승 시 승리 1회로 계산
* 콤보 발동 후 목표값은 다시 3~6 사이 랜덤으로 재설정
* 콤보 발동 후 combo count와 플레이 게임 타입 기록 초기화

## 콤보 보너스

* 초성퀴즈와 훈민정음을 둘 다 플레이한 상태에서 콤보 발동: +20%
* 한 게임만 반복한 상태에서 콤보 발동: +10%

## 플레이 타입 기록

* 초성퀴즈 승리 시 combo_played_chosung = true
* 훈민정음 우승 시 combo_played_hunmin = true

완료 기준:

* 유저별 combo target이 3~6 사이로 관리되어야 합니다.
* 콤보 발동 후 새로운 target이 다시 부여되어야 합니다.
* 두 게임을 섞어서 플레이한 경우 +20%, 한 게임만 반복한 경우 +10%가 적용되어야 합니다.

# 작업 단위 10. 랜덤 콤보 기능 구현

랜덤 콤보는 유물 기능입니다.
보상 정산 시점에 확률적으로 발동합니다.

## 랜덤 콤보 기본 규칙

* 발동 시 +20% 보너스
* 일반 콤보와 중첩 불가
* 일반 콤보와 랜덤 콤보가 동시에 발동되면 더 높은 보너스 하나만 적용
* 현재는 둘 다 최대 +20%이므로 동시에 발동되어도 +20%만 적용

## 랜덤 콤보 확률

1성:

* 기본 발동률: 0.3%
* 강화 1강당 추가 발동률: 0.034%
* 최대 발동률: 2%

2성:

* 기본 발동률: 2%
* 강화 1강당 추가 발동률: 0.025%
* 최대 발동률: 4%

3성:

* 기본 발동률: 4%
* 강화 1강당 추가 발동률: 0.04%
* 최대 발동률: 8%

## 일일 발동 제한

* 1성: 하루 최대 3회
* 2성: 하루 최대 5회
* 3성: 하루 최대 8회

날짜 기준:

* KST 기준 일일 초기화

완료 기준:

* 보상 정산 시점마다 랜덤 콤보가 확률적으로 판정되어야 합니다.
* 일일 발동 제한을 초과하면 더 이상 발동되지 않아야 합니다.
* 일반 콤보와 중첩되지 않아야 합니다.

# 작업 단위 11. 최종 보상 계산 함수 구현

초성퀴즈와 훈민정음에서 공통으로 사용할 보상 계산 로직을 구현해 주세요.

입력값:

* user_id
* baseReward
* gameType
* isWin

처리 내용:

1. 유저 유물 조회
2. 유물 보너스율 계산
3. 일반 콤보 발동 여부 확인
4. 랜덤 콤보 발동 여부 확인
5. 적용할 콤보 보너스율 결정
6. totalBonusRate = artifactBonusRate + comboBonusRate
7. rawBonus = baseReward × totalBonusRate
8. integerBonus = rawBonus의 정수 부분
9. decimalBonus = rawBonus의 소수점 부분
10. decimalBonus를 bonus_remainder에 누적
11. bonus_remainder가 1 이상이면 정수 부분을 추가 보너스로 지급
12. finalReward = baseReward + integerBonus + remainder에서 나온 추가 보너스
13. 포인트 지급
14. total_earned_point 갱신
15. 포인트 로그 저장

완료 기준:

* 초성퀴즈와 훈민정음이 같은 보상 정산 함수를 사용할 수 있어야 합니다.
* 보너스 소수점이 누적되어야 합니다.
* 포인트 증가와 bonus_remainder 변경은 transaction으로 처리되어야 합니다.

# 작업 단위 12. 유물 배틀 기능 구현

명령어 예시:

* `/유물배틀 @상대`

유저가 다른 유저에게 배틀을 신청하고, 승자는 패자의 포인트 일부를 가져옵니다.

## 배틀 기본 규칙

* 배틀은 포인트를 새로 생성하지 않습니다.
* 승자가 패자의 포인트 일부를 가져옵니다.
* 배틀 신청비는 시스템에 소각됩니다.
* 상대가 온라인이 아니어도 배틀 가능
* 자기 자신에게 배틀 불가

---

## 전투력 계산

1성:

* 기본 전투력: 100
* 강화 1강당 +5

2성:

* 기본 전투력: 450
* 강화 1강당 +10

3성:

* 기본 전투력: 1500
* 강화 1강당 +18

예시:

* 1성 50강 = 350
* 2성 80강 = 1250
* 3성 100강 = 3300

—

## 승률 계산

기본 승률:

* 50%

전투력 차이에 따른 변화:

* 전투력 차이 100당 약 5%p 변화
* 최소 승률: 20%
* 최대 승률: 80%

예시:

* 전투력 차이 0: 50%
* 전투력 차이 +100: 약 55%
* 전투력 차이 +300: 약 65%
* 전투력 차이 +500: 약 75%
* 전투력 차이 +600 이상: 최대 80%

—

## 배틀 신청비

신청자의 유물 성급 기준:

* 1성: 100P
* 2성: 500P
* 3성: 1500P

신청비는 승패와 관계없이 반환하지 않습니다.

—

## 배틀 약탈량

패자의 유물 성급 기준으로 계산합니다.

패자가 1성일 경우:

* 패자 보유 포인트의 2%
* 최소 50P
* 최대 800P

패자가 2성일 경우:

* 패자 보유 포인트의 3%
* 최소 200P
* 최대 3000P

패자가 3성일 경우:

* 패자 보유 포인트의 4%
* 최소 500P
* 최대 10000P

단, 패자의 현재 포인트보다 많이 빼앗을 수 없습니다.

---

## 배틀 제한

* 유저 1명당 하루 최대 배틀 10회
* 같은 상대를 하루 최대 3회 공격 가능
* 상대 보유 포인트가 1000P 미만이면 공격 불가
* 신청자의 포인트가 신청비보다 적으면 배틀 불가
* 자기 자신 공격 불가

---

## 배틀 처리 순서

1. 신청자 조회
2. 상대 유저 조회
3. 자기 자신 공격 여부 확인
4. 양쪽 유물 조회, 없으면 기본 유물 생성
5. 배틀 일일 제한 확인
6. 같은 상대 공격 제한 확인
7. 상대 포인트가 1000P 이상인지 확인
8. 신청자 포인트가 신청비 이상인지 확인
9. SQL transaction 시작
10. 신청자 포인트에서 신청비 차감
11. 양쪽 전투력 계산
12. 신청자의 승률 계산
13. 랜덤 판정으로 승패 결정
14. 패자의 포인트 기준 약탈량 계산
15. 승자 포인트 증가
16. 패자 포인트 감소
17. 양쪽 배틀 통계 업데이트
18. 같은 상대 공격 횟수 업데이트
19. 포인트 로그 저장
20. 배틀 로그 저장
21. transaction commit
22. 결과 메시지 출력

완료 기준:

* 배틀 신청비는 반드시 소각되어야 합니다.
* 패자의 포인트는 음수가 되면 안 됩니다.
* 약탈량은 패자의 현재 포인트를 초과하면 안 됩니다.
* 모든 포인트 변경은 transaction으로 처리되어야 합니다.

# 작업 단위 13. 유물 랭킹 기능 구현

명령어 예시:

* `/유물랭킹`

정렬 기준:

1. 성급 높은 순
2. 강화 수치 높은 순
3. 전투력 높은 순
4. 누적 획득 포인트 높은 순

랭킹 점수 계산이 필요하다면 다음 공식을 사용합니다.

* rankingScore = 성급 × 100000 + 강화 수치 × 1000 + 전투력

완료 기준:

* 2성 1강은 1성 50강보다 높은 순위여야 합니다.
* 3성 1강은 2성 80강보다 높은 순위여야 합니다.
* 랭킹에는 유저명, 성급, 강화 수치, 전투력, 보너스율을 표시합니다.

# 작업 단위 15. 예외 처리

반드시 처리해야 할 예외:

1. 유물이 없는 유저

   * 기본 유물 생성 또는 안내 메시지 출력

2. 포인트 부족

   * 강화, 성급강화, 배틀 진행 불가

3. 최대 강화 상태

   * 일반 강화 불가
   * 성급강화 가능 조건이면 성급강화 안내

4. 1강에서 강화 실패

   * 1강 유지
   * 0강으로 내려가지 않음

5. 3성 100강

   * 최종 상태
   * 일반 강화와 성급강화 모두 불가

6. 자기 자신 배틀

   * 불가

7. 배틀 대상 포인트 부족

   * 상대 포인트 1000P 미만이면 공격 불가

8. 배틀 약탈량 초과 방지

   * 패자 보유 포인트 이상으로 약탈 불가

9. 포인트 음수 방지

   * 모든 포인트 변경 후 point >= 0 보장

10. 동시성 방지

* 강화, 성급강화, 배틀, 보상 정산은 SQL transaction 사용
* 필요한 경우 유저 row lock 사용

# 작업 단위 16. 테스트 시나리오

다음 케이스를 테스트해 주세요.

## 강화 테스트

1. 1성 1강 유저가 강화 실패 시 1강 유지
2. 1성 49강 유저가 성공 시 50강 도달
3. 1성 50강 유저는 일반 강화 불가
4. 포인트 부족 시 강화 불가
5. 강화 성공/유지/실패 로그 저장 확인

## 성급강화 테스트

1. 1성 49강은 성급강화 불가
2. 1성 50강은 2성 도전 가능
3. 1성 성급강화 성공 시 2성 1강
4. 1성 성급강화 실패 시 1성 30~39강 보상 지급
5. 2성 80강 성공 시 3성 1강
6. 3성은 성급강화 불가

## 보상 테스트

1. 초성퀴즈 랜덤모드 기본 100P 유지
2. 초성퀴즈 주제선정모드 기본 50P 유지
3. 유물 보너스 적용 확인
4. bonus_remainder 누적 확인
5. 일반 콤보 +10%, +20% 적용 확인
6. 랜덤 콤보 +20% 적용 확인
7. 일반 콤보와 랜덤 콤보 중첩 불가 확인

## 훈민정음 테스트

1. 단어 정답 보상 5~11P 랜덤 지급
2. 단어 정답 보상 합계 최대 150P 제한
3. 우승 보너스 최대 250P 제한
4. 우승자가 아닌 유저는 우승 보너스 미지급
5. 유물 보너스와 콤보 보너스 적용 확인

## 배틀 테스트

1. 자기 자신 공격 불가
2. 상대 포인트 1000P 미만이면 공격 불가
3. 신청비 부족 시 배틀 불가
4. 신청비는 승패와 관계없이 소각
5. 승자에게 약탈 포인트 지급
6. 패자의 포인트 차감
7. 패자 포인트 음수 방지
8. 하루 배틀 10회 제한
9. 같은 상대 하루 3회 제한
10. 배틀 로그 저장 확인
